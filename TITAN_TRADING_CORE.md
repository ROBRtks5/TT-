## 1. ФИЛОСОФИЯ: "ТРУБОПРОВОД ПРАВДЫ" (PIPELINE OF TRUTH)

Торговая система не должна быть набором разрозненных реакций. Это должен быть строгий однонаправленный поток данных и решений.

**ИДЕАЛЬНЫЙ ЦИКЛ (THE TICK):**
1.  **SENSE (Восприятие):** Получение данных (Свечи, Стакан, Баланс) -> Нормализация -> Запись в State.
2.  **THINK (Анализ):** Расчет индикаторов -> Проверка Стратегии -> Генерация "Намерения" (Intent).
3.  **VALIDATE (Проверка):** Намерение проходит через Риск-Менеджер и Калькулятор Маржи. Хватает ли денег? Разрешено ли это сейчас?
4.  **ACT (Исполнение):** Отправка приказа на биржу -> Ожидание ответа -> Обновление State.
5.  **VERIFY (Сверка):** Проверка, что биржа подтвердила действие.

## 2. МАТЕМАТИКА КАПИТАЛА (POWER CALCULATION)

Самая частая ошибка — рассинхрон между "Виртуальным кошельком" робота и реальным счетом.

### 2.1. Понятия
*   **Liquid Portfolio:** Весь портфель (Кэш + Акции).
*   **Initial Margin (StartingMargin):** Средства, заблокированные биржей под *уже открытые* позиции.
*   **Open Orders Cost:** (Локальная переменная!) Средства, которые мы *планируем* заблокировать под лимитки, которые *еще не исполнены*.
*   **Free Margin (Чистый остаток):** `LiquidPortfolio - InitialMargin`.

### 2.2. Формула Покупательской Способности (Long Power)
Робот должен считать так:
`EffectivePower = (FreeMargin / RiskRateLong) - (ActiveGridOrders * Price * LotSize)`

*   **RiskRateLong:** Ставка риска лонга (обычно ~0.14 для плеча x7). Если API дает 0 или null, используем Fallback (1.0).
*   **Safety Buffer:** Всегда умножаем результат на `0.95` (5% буфер на проскальзывание и волатильность маржи).

### 2.3. Идемпотентность Бюджета
Перед *каждым* отдельным ордером в цикле (например, при расстановке сетки) мы должны вычитать его стоимость из `EffectivePower` локально, не дожидаясь ответа API обновления портфеля.

## 3. УПРАВЛЕНИЕ ОРДЕРАМИ (ORDER LIFECYCLE)

### 3.1. Атомарность Входа (Market Entry)
1.  **Check:** Есть ли уже позиция? (Блокировка повторного входа).
2.  **Calc:** Расчет лота `(Power * Aggression) / Price`. Проверка `MinLot`.
3.  **Lock:** Установка флага `isExecutionInProgress = true`.
4.  **Exec:** Отправка `PostOrder`.
5.  **Ack:** Получение `orderId`.
6.  **Sync:** Принудительное обновление Портфеля и Позиций.
7.  **Unlock:** Снятие флага.

### 3.2. Управление Сеткой (Grid Sync)
Сетка — это не хаос. Это строгая структура.
1.  **Virtual Grid:** Рассчитываем в памяти, где *должны* стоять ордера.
2.  **Actual Grid:** Получаем список активных ордеров с биржи.
3.  **Diff:** Сравниваем.
    *   Лишние (цена ушла, дрейф) -> `CANCEL`.
    *   Недостающие -> `PLACE`.
4.  **Sequencing:** Нельзя слать 10 ордеров `Promise.all`. Биржа даст 429 или ошибку "Not enough assets" из-за гонки проверок маржи.
    *   *Решение:* Sequential Execution (по одному) или Batching с локальным пересчетом бюджета.

## 4. СИНХРОНИЗАЦИЯ И ТАЙМИНГИ (SYNC & TIMING)

### 4.1. Boot Sequence (Запуск)
1.  **Silence:** Никаких торговых действий.
2.  **Fetch:** Загрузка Инструмента (FIGI, Lots).
3.  **Deep Sync:** Загрузка Аккаунта, Позиций, Активных Ордеров.
4.  **State Reconciliation:**
    *   Если есть поза -> Восстановить `entryPrice`, включить режим "Управление Позицией".
    *   Если есть "левые" ордера (не от сетки) -> Снять (Purge) или подхватить.
5.  **Stream Connect:** Подключение веб-сокета.
6.  **Ready:** Только теперь флаг `isBotActive = true`.

### 4.2. Watchdog (Сторожевой пес)
*   Если поток данных молчит > 60 секунд -> `STATUS: STALE_DATA`. Запрет открытия новых позиций.
*   Если `TimeNow` > `CloseTime` -> `STATUS: MARKET_CLOSED`. Принудительная остановка.

## 5. ЗАЩИТНЫЕ МЕХАНИЗМЫ (SAFETY GATES)

### 5.1. Стоп-Лосс Менеджер (Cerberus)
*   Стоп-лосс должен быть **Физическим** (Stop-Order на бирже), а не виртуальным.
*   **Sync Logic:** Каждые 10 секунд (или при изменении позы) проверяем:
    *   Есть поза? -> Должен быть стоп.
    *   Нет позы? -> Стопов быть не должно.
    *   Размер стопа != Размеру позы? -> `CANCEL` старый -> `PLACE` новый.

### 5.2. Sunset Protocol (Закрытие дня)
*   За 15 минут до клиринга/закрытия:
    *   Блокировка новых входов.
    *   Снятие всей сетки.
    *   Если есть поза -> Принудительное закрытие по рынку (Market Close).
    *   Проверка: Баланс в деньгах (Позиция = 0).

## 6. МЕТОДОЛОГИЯ ОТЛАДКИ (HOW TO FIX)

При проверке каждого модуля мы задаем вопросы:
1.  **Is it Async Safe?** Ждем ли мы ответа сервера перед обновлением локального стейта?
2.  **Is it Budget Safe?** Проверяем ли мы деньги перед отправкой ордера?
3.  **Is it Redundant?** Не делаем ли мы лишних запросов?
4.  **Is it Resilient?** Что будет, если на шаге 3 упадет Интернет? (Ответ: Система должна остаться в консистентном состоянии или уйти в Safe Mode).
